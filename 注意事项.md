# SingBox Client 注意事项

本文记录本轮调试中耗时较久、容易反复踩坑的问题。后续遇到“启动成功但无法访问网络”时，优先按这里的顺序排查。

## 1. 端口监听不等于代理可用

现象：

- Windows 系统代理显示已开启，例如 `127.0.0.1:7890` 或 `127.0.0.1:13443`。
- `Get-NetTCPConnection` 能看到 mixed inbound 端口处于 `Listen`。
- 浏览器和其他应用仍然无法访问任何网站。

结论：

- 监听只说明 sing-box inbound 打开了。
- 真实可用还取决于 DNS、路由、出站节点、系统代理是否被其他客户端覆盖。

建议排查：

```powershell
Get-NetTCPConnection -LocalAddress 127.0.0.1 -LocalPort 7890
Invoke-WebRequest -Uri https://www.baidu.com -Proxy http://127.0.0.1:7890 -UseBasicParsing -TimeoutSec 10
Get-Content "$env:USERPROFILE\.singbox-client\singbox-runtime.log" -Tail 120
```

## 2. DNS detour 不要硬编码旧节点

现象：

```text
start dns/tcp[google]: outbound detour not found: vless-reality
```

根因：

- DNS server 里保存了旧 profile 的 `detour: vless-reality`。
- 导入新订阅后，当前 outbounds 已经没有 `vless-reality` 这个 tag。

修复策略：

- 全局 settings 不保存远程 DNS 的固定 detour。
- 启动前根据当前 active outbound 动态写入 remote DNS detour。
- 如果找不到有效 outbound，则移除 remote DNS detour，避免启动直接 FATAL。

## 3. local DNS 必须直连，避免 DNS 死循环

现象：

```text
DNS query loopback in transport[local]
lookup h.gfwfw.com ...
```

根因：

- 节点服务器本身是域名，例如 `h.gfwfw.com`。
- 解析节点域名需要 local DNS。
- 但 local DNS 查询又被送进代理节点，代理节点又需要先解析域名，形成循环。

修复策略：

- local DNS 固定使用 `detour: direct`。
- remote DNS 才允许走当前代理节点。

推荐 DNS 结构：

```json
{
  "servers": [
    {
      "tag": "google",
      "address": "tcp://8.8.8.8"
    },
    {
      "tag": "local",
      "address": "223.5.5.5",
      "detour": "direct"
    }
  ],
  "final": "google"
}
```

## 4. 不要抢占正在使用的第三方系统代理

现象：

- 当前电脑已经有另一个代理软件在使用 `127.0.0.1:12334`。
- 本客户端启动后如果强制把 Windows `ProxyServer` 改成自己的端口，原有代理链路会被打断。
- 如果 sing-box 后端随后不可用，就会导致所有网站都无法访问，也无法继续通信。

修复策略：

- 启动前检测 Windows 当前系统代理。
- 如果当前已启用且不是本客户端的目标端口，则不覆盖系统代理。
- 只启动 sing-box 后端，并提示 `existing system proxy was left unchanged`。

## 5. rule-set 下载失败不能阻断基础启动

现象：

```text
initialize rule-set: Get "https://raw.githubusercontent.com/...": EOF
```

根因：

- 远程 rule-set 需要网络下载。
- 如果启动阶段网络还没建立，sing-box 可能因 rule-set 初始化失败而无法启动。

修复策略：

- 启动阶段先尝试 direct 下载 remote rule-set。
- 若失败或耗时过长，生成 bootstrap config，移除 remote rule-set 及引用规则后先启动。
- 基础代理能力优先于规则完整性。

## 6. 配置写入不要产生 UTF-8 BOM

现象：

```text
invalid character 'ï' looking for beginning of value: row 1, column 1
```

根因：

- Windows PowerShell 的某些写入方式会产生 UTF-8 BOM。
- sing-box JSON 解析不接受 BOM。

修复策略：

- 应用内部使用 Rust `fs::write` 写 JSON，避免 BOM。
- 手动脚本修复时使用 UTF-8 no BOM。

## 7. DNS strategy 的 UI 文案不能直接写进 sing-box

现象：

```text
dns.strategy: unknown domain strategy: IPv4 only
```

根因：

- UI 文案 `IPv4 only` 被直接写进配置。
- sing-box 只接受自己的枚举值；Auto 更接近“不写 strategy”。

修复策略：

- settings 中可显示 `auto`。
- 运行配置中遇到 `auto`、`IPv4 only`、`ipv4_only` 等旧写法时移除 `dns.strategy`。

## 8. 推荐排查顺序

1. 确认系统代理当前指向哪个端口。
2. 确认该端口是否真的有进程监听。
3. 用 `Invoke-WebRequest -Proxy` 直接测试本地代理。
4. 查看 runtime log 中第一条 `ERROR`，优先看 DNS。
5. 检查 DNS server 的 `detour` 是否合理。
6. 检查 active outbound 是否存在且能连接。
7. 若当前正在使用其他代理软件，先确认本客户端是否覆盖了 Windows `ProxyServer`。
